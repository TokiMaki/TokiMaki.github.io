# AGENTS.md

## 1. 기본 원칙

- 기본 응답은 한국어로 작성하고, 사용자의 직접 질문에 먼저 답한다.
- 리뷰·분석 요청에서는 구현을 명시하지 않은 한 코드를 수정하지 않는다.
- 구현 요청에서는 필요한 최소 범위만 수정한다.
- 요청과 무관한 리팩터링, 파일 이동, 스타일 정리, 구조 변경은 하지 않는다.
- 확실하지 않은 내용은 추측하지 말고 코드와 실행 결과를 확인한다.
- 기존 미커밋 변경사항을 임의로 되돌리거나 덮어쓰지 않는다.
- 비밀키, 토큰, 개인정보를 코드·로그·응답에 노출하지 않는다.

## 2. 프로젝트와 현재 방향

이 프로젝트는 **던파일럿(DunPilot)** 이다.

던전앤파이터 캐릭터의 현재 장비와 세팅을 분석하고, 가능한 스펙업 후보를 비교해 **골드 대비 효율이 높은 스펙업 순서**를 추천하는 서비스다.

> 핵심 질문: 현재 캐릭터에서 어떤 스펙업에 먼저 골드를 쓰는 것이 가장 효율적인가?

주요 범위는 네오플 API 기반 캐릭터 조회, 스펙업 후보 탐색, 딜·버프 상승량 계산, 경매장 가격·재료비 반영, 추천 row 생성과 정렬이다.

과거의 **헬 운영 최적화, 계시, 결정 조각, 솔리드 소울, 정가, 초월, 졸업 기대값**은 현재 기본 도메인이 아니다.

- `Docs/legacy/DNF_HELL_OPTIMIZER_SPEC.md`
- `계시 환산.xlsx`

위 자료는 사용자가 과거 기능을 명시적으로 요청한 경우에만 참고한다. 현재 스펙업 추천 로직의 근거로 자동 적용하지 않는다.

## 3. 목표 책임 경계

구조를 정리할 때 다음 책임 경계를 기준으로 한다.

- **API Client:** 네오플 API 호출과 응답 수신
- **Repository / Cache:** 캐시, TTL, single-flight, 묶음 조회, 실패 fallback
- **Candidate Resolver:** 비교 후보 탐색, 검색 조건, 후보 필터링
- **Calculator:** 딜, 버프, 비용 대비 효율 등 수치 계산
- **Presenter:** 프론트용 recommendation row와 payload 조립
- **Route:** HTTP 요청 파싱, 서비스 호출, 응답과 오류 코드 반환

현재 코드에 이 구조를 한 번에 강제하지 않는다.

- 한 번에 하나의 도메인만 분리한다.
- 클래스 계층이나 공통 인터페이스부터 만들지 않는다.
- 기존 흐름을 유지하면서 함수 이동 중심으로 최소 변경한다.
- 계산식과 추천 정책을 파일 분리 과정에서 다시 작성하지 않는다.
- 단순한 코드를 패턴이나 추상화로 감싸지 않는다.

## 4. 작업 시작과 탐색

코드 구현, 수정, 구조 검토를 시작할 때 `Docs/WORK_CONTEXT.md`가 있으면 먼저 `Active Snapshot`만 확인한다.

`src/dnfHellTool/enchantView.js` 분리 작업은 시작하거나 재개하기 전에 `Docs/architecture/ENCHANT_VIEW_REFACTOR_GOAL.md`를 반드시 읽고 목표·불변 조건·완료 기준을 다시 확인한다.

- 매번 문서 전체나 과거 로그를 읽지 않는다.
- 단순 질문이나 코드 변경이 없는 짧은 답변에서는 확인하지 않아도 된다.
- 파일이 없고 실제 구현 또는 구조 작업을 시작하는 경우에만 아래 최소 형식으로 생성한다.

```md
# WORK_CONTEXT

## Active Snapshot

- 현재 목표:
- 현재 주요 파일:
- 최근 변경:
- 중요한 결정/제약:
- 알려진 문제:
- 다음 작업:
```

코드 탐색은 다음 원칙을 따른다.

- 관련 함수명, 변수명, API 경로, payload 필드, UI 문구부터 검색한다.
- 검색 결과 주변 코드와 호출부를 확인한 뒤 필요한 경우에만 파일 전체를 읽는다.
- 관련 테스트, fixture, mock, 저장 응답이 있는지 확인한다.
- 중요한 호출부나 부작용을 놓칠 정도로 탐색 범위를 제한하지 않는다.

`Docs/QUESTION_LOG.md`는 자동으로 읽거나 생성·갱신하지 않는다. 사용자가 과거 결정 확인을 요청하고 파일이 있을 때만 필요한 부분을 확인한다.

레포지토리 전체 코드 리뷰에서는 `Docs/architecture/DUNPILOT_CODE_REVIEW.md`가 존재하면 해당 기준을 함께 따른다.

## 5. 수정과 코드 품질

- 작은 diff를 우선하고 요청받은 문제에 필요한 코드만 수정한다.
- 프로젝트의 기존 네이밍, import, 타입 힌트, 비동기 처리 스타일을 따른다.
- 함수와 변수 이름은 역할이 드러나게 작성한다.
- 중첩과 중복을 줄이되 이를 이유로 작업 범위를 넓히지 않는다.
- 의미 없는 임시 변수, 과도한 한 줄 표현, 불필요한 주석을 남기지 않는다.
- 주석은 코드가 무엇을 하는지보다 왜 필요한지를 설명할 때만 사용한다.
- 사용하지 않는 import, 변수, 죽은 코드를 새로 만들지 않는다.
- 관련 없는 파일의 포맷이나 줄바꿈을 대량 변경하지 않는다.

사용자의 명시적 요청 없이는 다음을 변경하지 않는다.

- 계산식과 추천 기준
- 후보 필터링 조건
- 딜러/버퍼 분기
- 추천 개수와 정렬 순서
- payload 구조, 필드 의미, null 처리
- 가격과 재료비 단위
- 프론트 구현
- 저장 데이터 구조
- 외부 라이브러리와 프레임워크

코드에 근거가 없는 효과, 가격, 확률, 보정값을 임의로 만들거나 추정해서 넣지 않는다.

## 6. 동작 보존 리팩터링

구조 분리나 함수 이동은 가능하면 다음 순서로 진행한다.

1. 변경 전 동일 입력의 응답 또는 fixture를 확보한다.
2. 대상 도메인 하나의 함수만 이동한다.
3. 기존 호출부를 최소 수정한다.
4. 같은 입력으로 재실행한다.
5. `debug`와 timing 값을 제외하고 결과를 비교한다.
6. 값, 개수, 정렬이 같을 때만 완료로 판단한다.

비교 대상에는 추천 개수·정렬 순서와 다음 핵심 필드가 포함된다.

`sourceType`, `slot`, `tier`, `itemName`, `effects`, `expectedGold`, `auction.minUnitPrice`, `materials`, `skillDamageMultiplier`, 서버가 제공하는 경우 `incrementalDamagePercent`

실시간 경매장 가격으로 결과가 흔들리면 저장 응답, mock 또는 fixture를 사용한다.

결과가 달라졌을 때 계산식이나 추천 기준을 바꿔 억지로 맞추지 않는다. 함수 이동, 입력 데이터, 캐시, 정렬 안정성, 누락된 의존성을 먼저 확인한다.

## 7. 검증

변경 범위에 맞는 가장 좁고 신뢰할 수 있는 검증을 실행한다.

1. 문법 또는 타입 오류 확인
2. 변경 함수와 호출 흐름 확인
3. 관련 단위 테스트 또는 fixture 비교
4. payload 값, 추천 개수, 정렬 순서 확인
5. 필요한 경우에만 넓은 통합 검증

- 기존 formatter, lint, test 명령이 있으면 변경 범위에 실행한다.
- 요청하지 않은 대규모 전체 테스트나 불필요한 외부 API 호출은 하지 않는다.
- UI를 변경했거나 사용자가 화면 확인을 요청한 경우에만 브라우저/MCP 검증을 우선 고려한다.
- 검증할 수 없는 부분은 이유와 범위를 명확히 적는다.
- 실행하지 않은 검증을 통과했다고 말하지 않는다.

## 8. WORK_CONTEXT와 문서 관리

`Docs/WORK_CONTEXT.md`는 현재 작업 상태를 다음 세션에 넘기기 위한 짧은 상태판으로 사용한다.

다음 경우에만 `Active Snapshot`을 갱신한다.

- 기능 구현 또는 중요한 버그 수정
- 계산식, 추천 기준, payload 계약 변경
- 구조 분리나 주요 파일 역할 변경
- 중요한 사용자 결정 반영
- 알려진 문제 또는 다음 작업 방향 변경

다음 경우에는 갱신하지 않는다.

- 단순 질문과 설명
- 코드 변경 없는 리뷰
- 사소한 포맷 수정
- 이미 기록된 내용의 반복

갱신 원칙:

- 기존 `Active Snapshot`을 최신 상태로 덮어써서 유지한다.
- 항목별 1~2줄로 짧게 쓰고 상세 작업 일지를 누적하지 않는다.
- 현재 목표, 주요 파일, 최근 변경, 중요한 결정/제약, 알려진 문제, 다음 작업만 남긴다.
- 민감정보, 긴 코드, 디버그 로그, 일회성 시행착오는 기록하지 않는다.
- `Session Log`는 사용자가 명시적으로 요청하지 않는 한 만들거나 누적하지 않는다.

과거 헬 최적화 문서는 삭제하지 않더라도 현재 기능 문서와 명확히 구분한다.
지속적으로 필요한 계산 기준, payload 계약, 중요한 예외만 별도 문서화한다.

## 9. 종료 응답

최종 응답에는 짧게 포함한다.

- 답변하거나 변경한 내용
- 변경한 주요 파일
- 실제로 수행한 검증과 결과
- 확인하지 못한 사항 또는 남은 위험
- 다음 작업이 꼭 필요하면 한 가지만 제안
